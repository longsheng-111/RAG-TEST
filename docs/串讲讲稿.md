## 1 · 功能走查 · 知识问答

【预计用时：12 min｜17:03-17:15｜屏幕状态：App 知识问答页（Docker 栈 localhost:3000）】

> 本章纪律：工程决策每个只讲 30 秒，听众感兴趣的自然会在第 4 章追问——**功能演示不变成技术讲座**。

### 站点 1.1 · 侧栏与会话列表

- 【演示操作】指左侧窄栏四个菜单（知识问答/知识库管理/上传文件/文件管理）→ 指会话列表两张存档卡（存档01 带"模拟面试"标签、存档02）。
- 【口播讲词】
  > "界面世界观是一本作业本：会话叫'存档'，每张卡上有槽位编号、人设贴纸、消息数和 token 消耗。注意卡片左下角这个数字——100,188 tokens——token 在这个系统里是一等公民，每一轮对话都计量、持久化。第 4 章我会讲它是怎么算的、超了怎么办。"

### 站点 1.2 · 新建会话弹窗

- 【演示操作】点会话栏右上角红色"+"→ 弹窗出现：标题"新建会话"、上方两个选项卡（知识问答/模拟面试）、人设选择、知识库选择 → 选"知识问答"、知识库选 knowledge_chunks → 创建。
- 【口播讲词】

- 【站点工程决策】（30 秒）创建动作本身不调 LLM，秒回；首轮提问后会用一条轻量 LLM 调用自动生成会话标题——标题不是手起的，是内容驱动的。【证据：routes.py:483-488，title_generator.py:21-23】

### 站点 1.3 · 空态

- 【口播讲词】

### 站点 1.4 · 现场提问（live，LLM 等待 10-30 秒）

- 【演示操作】在底部输入框输入预置问题 **"如何合并同事的 Git 分支？"** → 按 Ctrl+Enter 发送 → 等待期间不冷场，口播下方内容。
- 【口播讲词】（等待期填充）
  > "等待的这十几秒正好说这个系统现在最大的体验短板：全链路同步阻塞，没做 SSE 流式——检索、生成、评分都是一次性 JSON 返回。这是我在'不足'一章会正式认领的第一条。等待期间后台在干什么？先过一道守门链（输入校验、token 预算预检），然后检索——注意左边马上会出现一个检索面板。"
- 【站点工程决策】（30 秒）守门链叫 harness：每一轮问答都走同一条 DAG——校验 → 预算 → 检索（10s 超时）→ 空结果兜底 → 生成（30s 超时）。任何节点失败都有兜底回答，不会把异常栈抛给用户。【证据：harness.py:167-263】
- 【超时可砍】若 LLM 超过 30 秒未返回或报错：放弃 live，口播"现场网络不给力，看历史存档里的同款问答"，切存档02 继续。错误卡本身也能讲一句兜底设计。【时间锚点：此站最晚 17:08 必须结束】

### 站点 1.5 · 检索面板（"找到的线索"）

- 【演示操作】指左侧检索面板：标题"找到的线索"、来源卡（Git命令参考手册.md、83% 相似度、章节路径"Git 命令参考手册 > 9. 合并与变基 > 合并（merge）"）、"共 5 条"。
- 【口播讲词】
  > "回答之前先看证据。左边这个面板是本次检索的 5 个片段：文件名、相似度、还有章节路径。注意这个路径——每个切片都带它在原文里的章节位置，这是分块时保留的元数据，不是检索时才算的。"
- 【站点工程决策】（30 秒）检索不是单向量检索：BM25 关键词和向量语义双路召回，RRF 融合，再用同一个 BGE 模型精排，最后 0.65 相似度阈值过滤。为什么双路——专有名词（React、Git、BGE）纯向量召回差，这是八股文场景的致命伤。深展开在第 4 章。【证据：retrieval.py:63-67, 85-89, 91-117, 119-149】

### 站点 1.6 · 带引用的答案（"讲解员说"）

- 【演示操作】指答案区：标题"讲解员说"、token 数、"引用 5 条来源"、正文中的彩色圆形角标 ❶❷❸。
- 【口播讲词】
  > "答案区的每个事实陈述后面都跟着一个彩色角标——这就是 `[^N^]` 引用标注，颜色对应不同来源。AI 说的每句话，我都要求它标明出处。这个要求写在 prompt 里：引用具体来源时必须使用上标格式标注。"
- 【站点工程决策】（30 秒）引用是 prompt 层约束 + UI 层渲染的组合：模型按格式输出，前端把 `[^N^]` 解析成可交互元素并和检索面板的双向绑定。诚实边界：没有后校验机制，模型可能标错——所以才有下一站的双向对照。【证据：qa.py:66-70】

### 站点 1.7 · 引用浮层（hover 预览 → 点击钉住）

- 【演示操作】hover 一个角标 → 浮层出现（原文摘要、来源文件名、相似度）→ 点击角标 → 浮层钉住（完整原文 + 操作区）→ 指左侧检索面板对应来源卡同步高亮。
- 【口播讲词】
  > "hover 是预览，点击钉住看完整原文。这是**可信输出的第二层防线：双向对照**——AI 引用的话，用户可以逐条和原文核对。左边面板也会同步高亮对应的来源卡，两个方向都能跳转。"
- 【站点工程决策】（30 秒）浮层两态（preview/pinned）锚定在角标上定位，320px 固定宽、钉住态滚动。顺手说个真实事故：这个浮层上周刚修过一个定位 bug——外壳塌缩导致它溢出视口 160px，是响应式审计实测抓出来的。【证据：QAPanel.tsx:249-317，commit 9fe7c07】

### 站点 1.8 · 替换引用 / 标记不准确（纠错闭环）

- 【演示操作】切存档02 → 指答案底部"发现引用或结论有问题？"→ 点开"标记不准确"弹窗展示（不提交）→ 关上 → 指存档01/02 历史里的"已反馈"印章（如有）。
- 【口播讲词】
  > "这是整个系统我最得意的一环，**第三层防线：纠错闭环**。用户发现引用不对，可以'替换引用'或'标记不准确'。重点是这条反馈的去向：它不是写进日志等以后分析——它落盘后，**下一次对话时会被注入 system prompt 末尾**，原文是'以上约束优先级高于检索结果，后续回答请优先遵守'。也就是说，用户的纠正立刻变成后续回答的事实约束。"
- 【站点工程决策】（30 秒）注入点是 system prompt 不是 user prompt——约束等级不同。演示金句可直接用：**"这不是收集反馈等以后训练模型，是立刻生效的下一次对话约束。"**【证据：feedback.py:80-96，qa.py:99-103】

### 站点 1.9 · 工具栏 token 区

- 【演示操作】指顶部工具栏右侧：墨水分段条、"4,246 / 50,000"、清空对话按钮。
- 【口播讲词】
  > "回收开头埋的线：token 计量。这个分段条是本会话的'墨水用量'，4,246 / 50,000——50,000 是这个会话的 token 预算上限，到了就不让继续烧。优先用 API 返回的真实 usage，没返回时用估算兜底（中文 1 字约 1 token）。"
- 【站点工程决策】（30 秒）三层控制：可观测（每轮计量持久化）、有预算（单次输入 4000 / 输出 1024 / 会话 50000 三个硬上限）、有节约（累计超 60% 触发上下文压缩，早期对话压成摘要）。第 4 章讲压缩策略。【证据：token_tracker.py:81-95, 107-115, 152-194，config.py:96-98】

【转场句】

### 站点 2.4 · 评分反馈与低分追问

### 4.1 深潜 · 混合检索漏斗（1.75 min）

- **问题**：单一检索有盲区。纯向量对专有名词（React、BGE、Git 命令）召回差——训练语料里这些词不是高频上下文；纯关键词不理解语义——用户问"怎么优化首屏加载"，文档写的是"减少 First Contentful Paint"。
- **方案**：四级漏斗。BM25 召回 20 + 向量召回 20 → RRF 融合（`score = Σ 1/(60+rank)`，只依赖排名、不用校准分数尺度）取 Top 15 → 同一个 BGE 模型做 query×doc 交叉精排取 Top 5 → 相似度 < 0.65 丢弃，可以为 0 条触发兜底。
- **放弃项**：只做向量检索（八股场景致命伤）；Cross-Encoder 精排（效果更好但多一个模型，我用同一 BGE 双塔+交叉零额外依赖）；HNSW 近似搜索（数据量小，FlatIP 精确搜索足够快，改一行就能换）。
- **验证**：提问"React 和 Vue 的区别"，检查结果是否同时覆盖两个关键词；看响应里 `reasoning_steps` 的 top_score 与 confidence。
- 【证据：retrieval.py:63-67, 85-89, 91-117, 119-149, 151-167；config.py:103】

### 4.2 深潜 · 上下文工程（1.75 min）

- **问题**：上下文不是"切片扔进 prompt"。四个子问题：切片保不保留章节结构？多轮对话+检索+问题的拼装优先级？用户纠错怎么变成约束？面试和问答共用一套上下文吗？
- **方案**：① 分块带章节路径前缀（第 3 章讲过，不重复）；② Prompt 组装先算预算——`4000-1024-500=2476 tokens`，按预算逐条追加，超了截断，历史只留 6 条；拼装顺序固定：参考资料编号 → 对话历史 → 用户问题 → 回答要求（Markdown + `[^N^]` + 800 字内）；③ 纠错反馈格式化后注入 **system prompt 末尾**，"优先级高于检索结果"；④ 面试独立 exam_state，逐题线程隔离。
- **放弃项**：LangChain 全套编排（太重，我只借用了一个 `MarkdownHeaderTextSplitter`，其余自己写——可控性更好）。
- **验证**：打开 metadata.json 抽查 chunk 开头是否带标题路径；纠错前后问同一问题，看答案是否变化。
- 【证据：ingest.py:178-217；qa.py:27-38, 54-73, 99-103；feedback.py:80-96】

### 4.3 深潜 · Token 控制（1.75 min）

- **问题**：LLM 调用是计时收费资源，"能用"和"用得起、看得见"是两回事。
- **方案**：三层。可观测——优先 `response.usage` 真实数据，缺失时字符级估算兜底（中文 1 字≈1 token），会话级累计持久化。有预算——单次输入 4000 / 输出 1024 / 会话 50000 三个硬上限，超限由 harness 兜底返回提示、不再调 API。有节约——会话累计超 60%（30,000）触发上下文压缩：保留最近 4 轮完整对话，更早的调 LLM 压成摘要，LLM 不可用时降级为直接截断；prompt 里再要求 800 字以内输出。
- **放弃项/诚实边界**：tiktoken 精确计数没做——当前估算对中英混合偏保守（高估），接入要适配 DeepSeek 的 tokenizer，在 roadmap。
- **验证**：会话卡上的累计数字就是证据；压测一个长会话看 60% 压缩触发后的历史形态。
- 【证据：token_tracker.py:49-78, 81-95, 107-115, 152-194；harness.py:89-98；qa.py:70】

### 4.4 深潜 · 可信输出四层防线（1.75 min）

- **问题**：RAG 能减少幻觉不能消灭幻觉，系统设计上怎么把"可信"做到能力上限？

## 5 · FAISS 选型

【预计用时：4 min｜17:35-17:39｜屏幕状态：PPT 对比表页】

### 5.1 对比表（1 分钟）

| 维度 | FAISS（我的选择） | ChromaDB（常见替代） |
|------|------------------|---------------------|
| 本质 | C++ 向量计算库，Facebook 出品 | Python 向量数据库（包装 hnswlib） |
| 部署 | 零依赖，`pip install faiss-cpu` | 需要额外进程/服务 |
| 持久化 | `faiss.write_index()` → 二进制文件 | 自动持久化到 SQLite |
| 元数据 | 自己用 JSON 文件管理 | 内置 metadata 过滤 |
| 删除操作 | 不支持单条删除（整库重建） | 支持 `collection.delete(ids)` |
| 检索算法 | IndexFlatIP（精确内积） | 默认 HNSW（近似） |
| 适用规模 | < 10 万向量，精确搜索 | 百万级，近似搜索 |

### 5.2 决策逻辑（1.5 分钟，照念）

> "我的场景是单机、小规模、个人使用——当前 230 个 chunks、464KB 索引。这个前提下 FAISS 有三个关键优势：
>
> 第一，**精确搜索**。IndexFlatIP 暴力算全量内积，O(N)——但 N=230 的时候这根本不是问题。HNSW 近似搜索在极小数据量下反而可能漏掉最近邻。
>
> 第二，**零运维**。pip install 完就能用，没有额外服务进程。我的设计原则之一：不给用户添运维负担。
>
> 第三，**元数据可控**。ChromaDB 的 metadata 被锁在它的 schema 里；我自己用 JSON，随时改结构、加字段、做迁移。
>
> 代价我也清楚：删除要重建整个索引——`delete_file()` 就是收集保留文本→重新编码→建新索引→保存。230 chunks 下是毫秒级。"

【证据：vector_store.py:143, 315-349】

### 5.3 演进故事："目录叫 chroma_db 但从没用过 Chroma"（1 分钟，照念）

> "讲个这个项目里真实的技术史：仓库里曾经有个目录叫 `chroma_db`——但项目从头到尾没用过 ChromaDB，那是早期命名的遗留。后来统一改名 `vector_db`，代码里留了一个迁移函数：启动时自动探测旧目录（包括一个 BACKEND_DIR 算错一层导致的 `backend/backend/chroma_db`），找到就自动搬走，不丢数据。这个'命名腐烂'的故事下周还会再出现一次——在 Docker 配置里。"

【证据：config.py:27-62】

### 5.4 迁移触发条件（30 秒，照念）

> "我不是 FAISS 原教旨主义，迁移条件写死了两条：单库 chunk 超 20,000 或重建耗时超 10 秒——迁 Milvus Lite / ChromaDB；JSON 元数据超 5,000 条或出现多用户——迁 SQLite/PostgreSQL。选型是选'现在够用 + 知道何时换'，不是选永远。"

## 6 · Docker 部署实践

【预计用时：5 min｜17:39-17:44｜屏幕状态：终端 + docker-compose.yml】

### 6.1 为什么容器化（30 秒，照念）

> "两个动机。一是双端一致：我在 macOS 上开发，但部署目标是任意的——容器把'我机器上能跑'变成'哪里都能跑'。二是一键部署：`docker compose up -d` 一条命令拉起前后端，新机器不需要装 Python、Node、模型环境。"

### 6.2 三个真实漂移修复（2 分钟，本章叙事落点）

> "打开 docker-compose.yml 之前我以为它只是'有点旧'，查下来是三个时代的漂移叠在一起——**这是本章我想留下的观点：遗留配置与代码漂移，是最常见的工程腐烂，而且它不报错，只是安静地不对。**"

- **漂移一：compose 活在 ChromaDB 时代**（40 秒）
  > "卷名叫 `chroma_data`、环境变量叫 `CHROMA_PERSIST_DIR`——可代码早就迁到 FAISS 了。pydantic 配置对不上就静默忽略，意味着容器里的数据会写到根本没挂卷的地方，容器一删全丢。修法：卷改 `vector_data`、变量改 `VECTOR_PERSIST_DIR`，并补挂了会话数据卷——会话和反馈都落盘在 `data/` 下，之前完全没持久化。"
- **漂移二：`.env` 挂了个寂寞**（40 秒）
  > "compose 里把 `.env` 挂进容器，看起来天衣无缝。但读配置的代码从 config.py 上跳四级算项目根——在容器里算出来是根目录 `/`，它去读的是 `/.env`，根本不存在。结果：容器里没有任何 API key，LLM 全部走'未配置'分支，部署等于摆设。修法：compose 加 `env_file: .env` 直接注入进程环境，并用 `docker compose config` 验证解析结果。"
- **漂移三：只挂卷不给变量，等于没修**（40 秒）
  > "还有一层更隐蔽的：代码里的 BACKEND_DIR 解析为 `/app/app`，但卷挂在 `/app/data`、`/app/vector_db`——如果不显式给环境变量纠偏，应用会把数据写进 `/app/app/data`，完美绕过所有挂载。所以 VECTOR_PERSIST_DIR、SESSION_DIR 这些变量不是'配置项'，是把应用指回挂载点的纠偏器。"

【证据：docker-compose.yml；backend/app/core/config.py:11-14, 108-112】

### 6.3 部署架构走查（60 秒）

## 7 · 工程方法论：Loop + Harness 表里同构

【预计用时：4 min｜17:44-17:48｜屏幕状态：PPT + 终端 git log】

### 7.1 回收悬念：两个 loop、两层 harness（90 秒，照念）

> "开场我说系统里有两个 loop 和两层 harness。第一层在**产品代码**里：agent.py 的查询改写循环——检索不充分就让 LLM 改写查询再试，最多 3 轮，Jaccard 相似度超 0.8 判定死循环提前退出；harness.py 的守门链——每一轮问答都过同一条 DAG，校验、预算、超时、兜底，任何节点失败都有兜底回答，异常栈永远不到用户眼前。
>
> 第二层在**开发流程**里，结构一模一样：我的'检索'是把方案和事实基准对齐，不对就改写再来——STYLE_GUIDE 是 420 行的单一事实源，任何样式先登记文档再写代码；我的'守门链'是体检报告——259 行缺陷清单当任务注册表，每个修复项有优先级、有验收标准、有状态回写；我的'兜底'是一轮一 commit——每个 commit 单一主题、可独立 revert。
>
> 不追求一次做对，追求错了能被发现、被修正。**产品纠错回路与开发流程是同一套哲学。**"

【证据：agent.py:14-16, 100-116, 152-187；harness.py:167-263；docs/STYLE_GUIDE.md；docs/DX-RAG体检报告-2026-07.md】

### 7.2 流程实战案例：一次响应式审计（90 秒）

- 【口播讲词】
  > "说一个上周刚发生的完整循环。现象：页面在窗口化时布局溢出，两轮修复'没治好'。我没有直接再改第三轮，而是先做了一次实测审计——逐档视口量布局、核对每个媒体查询的量纲，最后发现真凶之一是**开发服务器在交付损坏的构建**：修复从未到达浏览器，'没治好'是假象。
  > 审计报告按严重度定级 F1-F5 交给项目负责人拍板，然后一轮一 commit：min-width 调整一个、超宽上限一个、内联像素收编一个、浮层定位根治一个、文档升级一个。每个 commit 配实测验收矩阵，最后 build 通过收尾。还把'溢出永不静默、定宽铬件唯一清单'写进了 STYLE_GUIDE 的布局不变量——**事故修完要变成规矩，不然白修**。"
- 【站点工程决策】（30 秒）这套"审计→定级→拍板→一轮一 commit→验收"的循环，就是 harness 哲学的流程版：每个节点有验证，失败有去处。【证据：docs/STYLE_GUIDE.md 4.12，commit 2333bdf/0366912/3840c68/9fe7c07/7753e5c】

### 7.3 真实提交记录（60 秒）

- 【演示操作】终端 `git log --oneline -8`，指着说。
- 【口播讲词】

## 9 · Q&A 附录（不占时长，10 分钟问答缓冲用）

> 沿用作战手册 Q1-Q10 骨架并更新至最新事实；新增 Q11。

**Q1 怎么控制幻觉？**
不是"控制"是"减少"：prompt 约束 `[^N^]` + 双向对照 UI + 纠错闭环。诚实边界：无后校验，依赖模型遵守 prompt；RAG 减少幻觉不消灭幻觉。【证据：qa.py:66-70，feedback.py:80-96】

**Q2 为什么 BGE-small 而不是 text-embedding-3-large？**
本地模型零 API 成本零网络延迟，512 维对 230 chunks 足够；API embedding 多 50-200ms 延迟加费用，个人项目不划算。上万级再考虑 BGE-large 或 API。【证据：config.py:86，vector_store.py:31-47】

**Q3 Chunk size 为什么 800？**
中文信息密度高：256 太短切碎知识点、1024 太长混入不相关内容、800 是一个完整知识点的实践折中。验证法：传已知八股文提问，看相关性——不相关多则调小，缺上下文则调大。【证据：config.py:92】

**Q4 Top-k 为什么 5？**
不是直接取 5：20+20 召回 → RRF 后 15 → 精排后 5。3 条有时缺信息，10 条噪声明显增加。【证据：retrieval.py:151-167】

**Q5 知识库外怎么拒答？**
两层：prompt 要求"资料不足明确说明"；检索层 0.65 阈值丢弃后可为 0 条，走兜底"知识库中暂无相关资料"。诚实边界：这是检索为空导致的兜底，不是模型主动判断"我不知道"。【证据：qa.py:67,88-89；retrieval.py:144-146；harness.py:106-114】

**Q6 为什么不用 ChromaDB/Milvus/Pinecone？**
见第 5 章完整对比表。一句话：单机小规模不需要向量数据库服务，FAISS 零依赖精确搜索够用，迁移触发条件已写死。【证据：第 5 章证据栏】

**Q7 全链路同步阻塞，为什么不加 SSE？**
现状确认未做（全仓无 StreamingResponse）。规划：先超时+阶段提示，再 SSE。不动技术栈无新依赖，但动 qa.py/agent.py 调用链，单独立项。【证据：routes.py:407,535,612】

**Q8 并发和多用户？**
当前无账号体系无并发处理，同步端点多用户会排队。短期 FastAPI async + 线程池缓解 I/O 等待；长期账号体系+数据库是前置。写入规划不属于当前迭代。【证据：体检报告远期规划】

**Q9 检索变好了怎么验证？评测指标？**
没有 RAGAS 没有标注集。验证目前是手工：改参数就提典型问题看相关性。TESTING.md 已设计回归方案（固定问句集 + Hit@k 断言 + 双形态 CI）未实现。最诚实的短板，但我知道它在哪且有补救方案。【证据：docs/TESTING.md:71-118】

**Q10 数据安全：文档存哪？API key 怎么管？**
文档在 uploads/，向量在 vector_db/，会话在 data/sessions/——Docker 里全部挂卷持久化。key 走 .env + compose `env_file` 注入，不进 git（.gitignore 可查）。传给 DeepSeek 的只是检索片段不是全量文档。路径穿越有 `_safe_file_path` 防护。【证据：routes.py:68-78；docker-compose.yml】

**Q11 Docker 化遇到的最大坑？（新增）**
三个，按疼度排序：① Next.js rewrites 是构建期内联，运行期 env 不生效——后端地址必须 build ARG 注入，卡了两轮重建；② `.env` 挂载路径与代码 PROJECT_ROOT 解析错位，key 根本没进容器，`docker compose config` 一眼看穿但事前没人看；③ 健康检查没有 start_period，与首次模型下载的慢启动竞态，90 秒误判后端死亡。共性教训：**配置漂移不报错，只是安静地不对——部署契约要和代码一样被审计。**【证据：frontend/Dockerfile:15-18；docker-compose.yml:21-30；backend/app/core/config.py:11-14】
