## 三、技术深潜 · 上下文工程

### 3.1 问题

RAG 系统的上下文不是"把文档切片扔进 prompt"就完事了。要解决四个问题：
1. 切出来的片段能不能保留原文的章节结构？
2. 多轮对话 + 检索结果 + 用户问题，拼装的优先级和截断策略是什么？
3. 用户说"这个引用不对"，这个信息怎么变成后续回答的约束？
4. 面试模式和问答模式共用一套 prompt 吗？

### 3.2 我的方案

**分块：标题感知的层级切分，不是定长硬切。**

很多 RAG demo 的做法是 `text[i:i+500]`，固定长度。我用的是一条三级切分管线：

1. **Markdown 标题切分**：用 `MarkdownHeaderTextSplitter` 按 `#/##/###/####` 标题分割，保留标题路径作为前缀。
   > 比如一篇"RAG 八股文"有 "## 一、RAG 基础概念" → "### 1. 什么是 RAG"，切出来的 chunk 开头是 `RAG 八股文 · 面试速查 > 一、RAG 基础概念 > 1. 什么是 RAG`
   
   【证据：`ingest.py:178-217`】

2. **段落切分**：标题块 ≤ 800 字符直接保留；超长的把标题前缀保留在每个子块中，按段落递归切分。
   
   【证据：`ingest.py:148-171`，`config.py:92` chunk_size=800】

3. **递归字符切分**：段落还超长就降级分隔符：`\n\n` → `\n` → `。！？；，` → 空格 → 硬切。重叠 120 字符保证边界不丢上下文。
   
   【证据：`ingest.py:220-273`，`config.py:93` overlap=120】

**Prompt 组装：先算预算，再拼内容。**

```
max_context = max_input_tokens(4000) - max_output_tokens(1024) - 500 = 2476 tokens
```

按这个预算，从检索结果中逐个追加 chunk，超了就截断。对话历史只保留最近 6 条。

【证据：`qa.py:27-38`】

User Prompt 的拼装顺序：
1. `## 参考资料` → 编号片段（`[参考片段 1]...`）
2. `## 对话历史` → 最近 6 条
3. `## 用户问题` → 原文
4. `## 回答要求` → Markdown + `[^N^]` 引用标注 + 800 字以内

【证据：`qa.py:54-73`】

**纠错回路：反馈不是日志，是事实约束。**

用户提交反馈后落盘到 `feedbacks.json`。下次生成回答时，`format_feedback_for_prompt()` 把所有历史反馈格式化为：

```
【历史反馈 / 事实修正约束】
1. 反馈类型：replace
   原引用片段：xxx
   应优先采用的片段：yyy
以上约束优先级高于检索结果，后续回答请优先遵守。
```

直接注入 **System Prompt 末尾**——不是 user prompt，是 system。这意味着模型在所有工具调用和回答生成中都受这个约束。

【证据：`feedback.py:80-96`，`qa.py:99-103`】

**面试模式上下文隔离：独立状态机，不污染 QA 上下文。**

面试用独立的 `exam_state` 结构，包含 `current_thread`（当前题目对话线程）、`reference_points`（参考答案要点）、`scores`（历史得分）。每次评分时只传当前题目的线程，不同题目之间上下文完全隔离。追问时传入完整 thread。

【证据：`routes.py:576-593`，`examiner.py:215-223`】

### 3.3 放弃的方案

- **固定长度切分**：最简单但会从句子中间切断，丢失语义边界。选标题感知切分的代价是依赖 Markdown 格式——纯文本文件效果差一些，但我的主要场景（八股文笔记）恰好都是 Markdown。
- **LangChain 全套**：太重。我只用了 `MarkdownHeaderTextSplitter` 这一个类，其余切分逻辑自己写的递归函数——可控性更好。

### 3.4 怎么验证

- 打开 `metadata.json`，抽查 3-5 个 chunk 开头是否带标题路径前缀
- 上传一个超过 800 字符的长段落 Markdown 文件，检查 chunk 数量和边界
- 提问后检查浏览器 Network 面板 `/api/chat` 返回的 `sources` 数组结构

## 四、技术深潜 · 混合检索漏斗

### 4.1 问题

单一检索方式有盲区。纯向量检索对专有名词（如 "React"、"BGE"）召回差，因为训练语料中这些词可能不是高频上下文；纯关键词检索不理解语义——用户问"怎么优化首屏加载"，文档里写的是"减少 First Contentful Paint"。

### 4.2 我的方案：四级漏斗

```
BM25 关键词召回 × 20
     ↘
向量语义召回 × 20  ──→ RRF 融合(k=60) → Top 15
                                           ↓
                                    BGE Rerank 精排 → Top 5
                                           ↓
                                    相似度 < 0.65 丢弃
```

**第一级：双路召回**
- BM25 召回 20 条（轻量中文分词：逐字切中文 + 连续字母数字切英文）
- 向量召回 20 条（BGE 编码 → FAISS IndexFlatIP 精确搜索）

【证据：`retrieval.py:63-67`，`retrieval.py:85-89`】

**第二级：RRF 融合**

公式 `score = Σ 1/(k + rank_i)`，k=60。不需要校准两种检索的分数尺度——只依赖排名。融合后取 Top 15。

```python
# 我就是这么写的
for rank, (file_name, content, _) in enumerate(bm25_results):
    scores[key] = scores.get(key, 0) + 1.0 / (60 + rank + 1)
```

【证据：`retrieval.py:91-117`】

**第三级：BGE Rerank 精排**

用同一个 BGE 模型对 15 个候选做 query × document 交叉相似度计算。这一步比召回阶段的相似度更精确——召回是双塔（分别编码 query 和 doc），Rerank 是交叉（query 和 doc 联合计算）。

【证据：`retrieval.py:119-149`】

**第四级：阈值过滤**

重排序后，相似度 < 0.65 的结果直接丢弃。这意味着检索结果可能少于 5 条，甚至为 0——触发兜底回答"知识库中暂无相关资料"。

【证据：`config.py:103`，`retrieval.py:144-146`】

### 4.3 放弃的方案

- **只做向量检索**：简单但专有名词召回差，这是做面试八股文场景的致命伤——"React"、"Vue"、"BGE" 这类术语恰好是检索关键词。
- **Cross-Encoder 做 Rerank**：效果更好但需要额外模型。我用同一个 BGE 模型做双塔召回 + 交叉精排——零额外依赖。
- **HNSW 近似搜索**：数据量小（<100 chunks），精确搜索 FlatIP 足够快。数据量上去了改一行代码换 IndexHNSWFlat。

### 4.4 怎么验证

- 提问 "React 和 Vue 的区别"，检查返回结果是否同时覆盖两个关键词
- 检查 `reasoning_steps` 中的 `top_score` 和 `confidence` 字段
- ⚠️核对：当前 index.ntotal 是多少（`ls backend/app/vector_db/knowledge_chunks/`）

## 五、技术深潜 · Token 控制

### 5.1 可观测

每次 API 调用都有 token 计量。优先用 `response.usage` 真实数据，API 没返回时用估算兜底：

```
估算公式：中文 1 字 ≈ 1 token，英文按空格分词 ≈ 1 token
```

【证据：`token_tracker.py:81-95`，`token_tracker.py:49-78`】

每个会话持久化 `total_tokens` 字段，前端可以展示——目前前端会话卡上展示了 token 数。

【证据：`session.py:33,45,92-93`】

### 5.2 有预算

三个硬上限：

| 预算项 | 值 | 位置 |
|--------|-----|------|
| 单次输入 | 4000 tokens | `config.py:96` |
| 单次输出 | 1024 tokens | `config.py:97` |
| 会话累计 | 50000 tokens | `config.py:98` |

上下文截断公式：
```
max_context = 4000 - 1024 - 500 = 2476 tokens
```

检索结果按此预算逐个追加，超出截断——不是"把 20 条结果全塞进去然后抱怨超限"。

【证据：`qa.py:27-38`】

超限行为：`check_token_budget()` 返回提示文字 → Harness 兜底返回，不会继续调 API。

```
"当前会话 Token 消耗已达上限（50000/50000），请新建会话继续提问。"
```

【证据：`harness.py:89-98`】

### 5.3 节约手段

**上下文压缩（已实现）**：当会话累计 token 超过 60% 阈值时触发。保留最近 4 轮完整对话，更早消息调用 LLM 生成摘要。LLM 不可用时降级为直接截断。

```
60% × 50000 = 30000 tokens → 触发压缩
```

【证据：`token_tracker.py:107-115`，`token_tracker.py:152-194`】

**Prompt 约束输出长度**：User Prompt 中明确要求"总字数控制在 800 字以内"。

【证据：`qa.py:70`】

### 5.4 还没做的（诚实列在 roadmap）

- **tiktoken 精确计数**：目前是字符级估算，对中英混合文本的估算偏保守（高估）。接入 tiktoken 可以做到精确到 token 的计数——但这需要适配 DeepSeek 的 tokenizer。
- **检索结果缓存**：同一 query 短时间内重复检索，目前每次都重新编码+搜索。对高频问句可以做缓存。

## 六、技术深潜 · 可信输出验证

### 6.1 四层防线

**第一层：Prompt 约束**

User Prompt 中明确：
> "如果参考资料不足以回答问题，请明确说明"
> "引用具体来源时，必须在对应事实陈述后使用 Markdown 上标格式 `[^N^]`"
> "回答要简洁、准确、有条理，总字数控制在 800 字以内"

【证据：`qa.py:66-70`】

**第二层：引用双向对照 UI**

前端渲染 `[^N^]` 为可点击的彩色圆形引用角标，点击后弹出引用浮层——展示原文片段 + 来源文件名 + 相似度分数。用户可以**逐条核对** AI 说的事实和原文是否一致。

（这是 QAPanel.tsx 的核心交互，不是你演示时说"有引用"就过去了——必须当场点一个角标给大家看。）

**第三层：纠错闭环**

用户发现引用不对 → 提交反馈（replace/inaccurate）→ 落盘 → 下次对话注入 system prompt → AI 优先遵守。

这不是"收集用户反馈等着以后训练模型"——是**立刻生效**的下一次对话约束。演示时可以当场验证：纠错前提一个问题，纠错后再提同一个问题，答案变了。

【证据：`feedback.py:80-96`，`qa.py:99-103`】

**第四层：检索回归测试（规划中）**

TESTING.md 设计了检索质量回归测试方案：固定问句集 + 真模型 + 真索引 → 每次改 embedding/切分参数后跑一遍，检查 top_k 命中率有没有下降。

目前零实现。但规划本身是工程思维的证据——我知道"改了 embedding 模型检索结果可能悄悄变差"这个风险，并且设计了验证方法。

【证据：`TESTING.md:71-81`】

### 6.2 系统局限（诚实声明）

1. **引用标注依赖模型遵守 prompt**：`[^N^]` 格式是 prompt 要求的，没有后校验机制——模型可能不遵守、可能标错编号。正确的做法是后处理正则提取+校验 N 是否在范围内，目前没做。
2. **无法检测"看起来合理但事实错误"的回答**：RAG 能减少幻觉但不能消灭幻觉。如果检索结果本身就有错误（文档写错了），模型会忠实地输出错误内容并标注引用——用户以为"有引用就是对的"，实际上源头就是错的。
3. **关键决策请核对原文**：不要因为 AI 引用了一段话就认为那段话一定正确。

## 七、技术深潜 · FAISS vs ChromaDB

### 7.1 一页对比

| 维度 | FAISS (我的选择) | ChromaDB (常见替代) |
|------|-----------------|-------------------|
| 本质 | C++ 向量计算库，Facebook 出品 | Python 向量数据库（包装了 hnswlib） |
| 部署 | 零依赖，`pip install faiss-cpu` | 需要额外进程/服务 |
| 持久化 | `faiss.write_index()` → 二进制文件 | 自动持久化到 SQLite |
| 元数据 | 自己用 JSON 文件管理 | 内置 metadata 过滤 |
| 删除操作 | 不支持单条删除（整库重建） | 支持 `collection.delete(ids)` |
| 检索算法 | IndexFlatIP（精确内积） | 默认 HNSW（近似） |
| 适用规模 | < 10 万向量，精确搜索 | 百万级，近似搜索 |

### 7.2 我的决策逻辑

"我的场景是单机、小规模（当前约 30 个 chunks，460KB 的 FAISS 索引）、个人使用。在这个前提下，FAISS 有几个关键优势：

第一，**精确搜索**。IndexFlatIP 是暴力计算所有向量的内积——O(N) 复杂度，但在 N=30 的时候这根本不是问题。ChromaDB 默认用 HNSW 近似搜索，在 N=30 的时候精确度反而可能不如 FlatIP——近似算法在极小数据量下有时会漏掉最近邻。

【证据：`vector_store.py:143` `faiss.IndexFlatIP`】

第二，**零运维**。FAISS 就是一个 C++ 库，`pip install` 完就能用。ChromaDB 需要额外的服务进程和持久化层。我的项目设计原则之一是'不给用户添运维负担'。

第三，**可控的元数据管理**。ChromaDB 内置 metadata 过滤很强，但也意味着元数据结构被锁在 ChromaDB 的 schema 里。我自己用 JSON 管理——虽然简陋，但我可以随时改数据结构、加字段、做迁移，不需要数据库 migration。

代价我也清楚：删除操作要重建整个索引。我的 `delete_file()` 函数就是收集保留文本 → 重新编码 → 新 IndexFlatIP → 保存。

【证据：`vector_store.py:315-349`】

当前数据量（30 chunks）下重建是毫秒级的。什么时候需要迁移？我设了两个触发条件：
- 单库 chunk > 20,000 或重建耗时 > 10 秒 → 迁向量数据库（Milvus Lite / ChromaDB）
- JSON 元数据文件 > 5,000 条或多用户需求出现 → 迁 SQLite / PostgreSQL"

## 八、表里同构：Loop + Harness 工程实践

> "产品纠错回路与开发流程是同一套哲学。"

### 8.1 产品层：Agent Loop + Harness 守门

**查询改写循环（ReAct 模式）**：

```
用户提问 → 检索 → 判断充分性(similarity ≥ 0.65?)
                      ↓ 不充分
                 LLM 改写查询 → 再检索 → 再判断
                      ↓ 充分           ↓ 3轮仍不充分
                   生成回答          取已有结果生成
```

最多 3 轮，每轮有循环检测（Jaccard > 0.8 判定为死循环，提前退出）。

【证据：`agent.py:14-16`，`agent.py:152-187`，`agent.py:100-116`】

**Harness 预算守门**：

每一次问答都经过 Harness 的 DAG：
```
输入校验(空/超长/敏感词) → Token 预算预检 → 检索(10s 超时)
→ 检索为空兜底 → 生成(30s 超时) → 格式化来源
```

任何节点失败都有兜底回答，不会把异常栈抛给用户——这一点在代码中被特意处理过（`routes.py:467-469` 的 500 隐藏内部细节）。

【证据：`harness.py:167-263`，`harness.py:59-71` 输入过滤，`harness.py:89-93` 预算检查】

### 8.2 流程层：风格基准 + 体检报告 + 提交纪律

**STYLE_GUIDE 作为单一事实源**：
407 行 Markdown，覆盖 token 定义、组件规范、文案指南、验收清单。新增任何样式/组件必须先登记 STYLE_GUIDE 再写代码。这不是"写完代码补文档"——是"文档不过审代码不写"。

【证据：`docs/STYLE_GUIDE.md`】

**体检报告作为任务注册表**：
259 行缺陷清单，按 P0/P1/P2 优先级排列。每轮迭代从报告里取修复项，修完回写状态。报告的"新发现区"用于记录过程中发现的、但本轮不改的问题——防止范围蔓延。

【证据：`docs/DX-RAG体检报告-2026-07.md`】

**真实提交记录（纪律证据）**：

```

af8b806 feat(classroom): v1.8 毛毡公告栏 + 粉笔盒拟真重绘 + 字体连贯性统一

```

注意 commit message 的规律：
- `feat(classroom):` / `fix(blackboard):` / `refactor:` —— 有前缀分类
- 每个 commit 单一主题——"一轮一 commit"
- 标题是解决方案摘要，不是"update code"这种废话

这意味着任何一个 commit 都可以独立 revert，不会出现"一个改动改了 17 个不相关文件"的情况。

### 8.3 收尾金句

> "这个项目的产品层有一个纠错回路——检索不充分就改写查询再试，用户说引用不对就注入 system prompt 修正。开发流程也有一条纠错回路——STYLE_GUIDE 是 spec，体检报告是 bug tracker，一轮一 commit 是 checkpoint。不追求一次做对，追求错了能被发现、被修正。产品纠错回路与开发流程是同一套哲学。"

## 九、面试模式深度

### 9.1 题型五轮换

考官不是随机出题——按固定顺序轮换题型：

```
第1题 → 概念题（基础定义）
第2题 → 原理题（底层机制）
第3题 → 对比题（辨析能力）
第4题 → 场景题（实战经验）
第5题 → 综合题（深入综合）
```

`question_index % 5` 取模轮换，保证每场面试覆盖完整的能力维度。

【证据：`examiner.py:275-276`】

### 9.2 评分：JSON 优先 + 正则兜底

评分输出优先解析 JSON：
```json
{"points": [...], "score": 7, "comment": "...", "supplement": "...", "correction": "...", "has_next_question": true}
```

JSON 解析失败 → 回退正则提取 `【评分】/【点评】/【补充】/【纠正】`。两层兜底——模型不输出 JSON 也不会崩。

【证据：`examiner.py:324-374`】

### 9.3 追问逻辑

评分 < 7 分 且追问次数未用完（最多 2 次）→ 追问。追问时传入完整 `current_thread`，考官针对漏洞追问，不换题。≥ 7 分或追问次数用完 → 进入下一题。

【证据：`routes.py:730-751`】

### 9.4 反作弊

两重检测：
1. **重复提交**：Jaccard 相似度 ≥ 0.85 判定为重复答案
2. **复制原文**：回答包含参考资料原文片段（30 字符连续匹配）

命中 → 评分 0 分 + 提示"请独立作答"

【证据：`examiner.py:160-196`，`examiner.py:18-19` 阈值常量】

### 9.5 答案缓存

基于 `MD5(题目 || 回答)` 缓存评分结果。同一题目 + 同一回答不会重复调用 LLM 评分。缓存存储在 `exam_state.answer_cache` 中。

【证据：`examiner.py:199-212`】

## 十、预设拷问 Q&A

### Q1: 怎么控制幻觉？

**答案骨架**：不是"控制"，是"减少"——Prompt 约束引用标注（`[^N^]`） + 前端引用对照 UI + 纠错闭环。但没有后校验机制，依赖模型遵守 prompt。诚实说：RAG 能减少幻觉但不能消灭幻觉。

【证据：`qa.py:66-70` 引用标注要求，`feedback.py:80-96` 纠错注入】

### Q2: 为什么选 BGE-small 而不是 text-embedding-3-large？

**答案骨架**：BGE-small 是本地模型，零 API 成本、零网络延迟。512 维足够区分当前知识库规模（30 chunks）。换 OpenAI embedding 会增加 50-200ms 网络延迟 + API 费用，对个人项目不划算。如果数据量到万级别，可以考虑换 BGE-large（1024 维）或 API embedding。

【证据：`config.py:86`，`vector_store.py:31-47` 本地加载】

### Q3: Chunk size 为什么是 800 而不是 256/512/1024？

**答案骨架**：800 是中文场景的实践值——中文信息密度比英文高。256 太短：一个问题可能被切成 3 块，检索只能抓到其中一块，上下文不完整。1024 太长：一个 chunk 包含多个不相关知识点，检索精度下降。800 是折中：一个完整的知识点（标题+内容）通常在这个范围内。

验证方法：上传一篇已知内容的八股文 → 提问 → 检查检索结果的 relevance。如果不相关 chunk 多 → 调小 size。如果答案缺上下文 → 调大 size。

【证据：`config.py:92`】

### Q4: Top-k 为什么是 5？

**答案骨架**：召回阶段 20+20→15→5，不是直接检索 5 条。最后 5 条是经过 RRF 融合 + BGE Rerank 精排的结果——5 条是"足够覆盖一个问题的知识点"与"不塞太多噪声"的平衡。实测：3 条有时缺信息，10 条噪声增加明显。

【证据：`retrieval.py:151-167`】

### Q5: 知识库外的内容怎么拒答？

**答案骨架**：两层。Prompt 层："如果参考资料不足以回答问题，请明确说明"。检索层：相似度 < 0.65 的结果直接丢弃，如果 Rerank 后 0 条结果，走兜底回答"知识库中暂无相关资料"。但这不是真正的"拒答"——是检索为空导致的兜底，不是模型自己判断"我不知道"。

【证据：`qa.py:67,88-89`，`retrieval.py:144-146`，`harness.py:106-114`】

### Q6: 为什么不用 ChromaDB / Milvus / Pinecone？

**答案骨架**：见第七章 FAISS vs ChromaDB 完整对比表。一句话：单机小规模不需要分布式向量数据库，FAISS 零依赖、精确搜索、够用。写了迁移触发条件（20,000 chunks / 10s 重建），不是永远不迁。

### Q7: 现在全链路同步阻塞，为什么不加 SSE？

**答案骨架**：现状确实没做——`/api/chat` 和 `/api/exam/*` 都是一次性 JSON 返回，grep 全仓无 `StreamingResponse`。这是体验层面的痛点：LLM 生成 10-30s 期间用户面对假死。

规划了但还没做。优先级：先做超时+阶段提示（"检索中…→生成中…"），再做 SSE 流式改造。流式改造不动技术栈、不需要新依赖，但要动 `qa.py`/`agent.py` 的调用链——需要单独一个迭代。

【证据：`routes.py:407,535,612`，体检报告 5.1】

### Q8: 并发和多用户怎么办？

**答案骨架**：当前单体架构无账号体系（全仓无 auth），无并发处理。`/api/chat` 和 `/api/exam/*` 是同步阻塞端点，多用户同时调用会排队。短期方案：FastAPI 本身支持 async，改 async/await + 线程池可以解决 I/O 等待阻塞。长期方案：账号体系 + 数据库是前置条件。写入规划但不属于当前迭代范围。

【证据：体检报告: "当前单体架构无账号体系"】

### Q9: RAG 的评测指标是什么？怎么验证"检索变好了"？

**答案骨架**：目前没有量化的评测指标——没有 RAGAS、没有人工标注测试集。验证方式：TESTING.md 设计了检索回归测试方案（固定问句集 + 基准结果）但零实现。

诚实的讲法："我的验证方式目前是手工——改了检索参数就提几个典型问题，检查返回结果是否相关。TESTING.md 写了自动化方案但还没实现。这是最诚实的短板——但至少我知道这个短板在哪，并且设计了补救方案。"

【证据：`TESTING.md:71-81`】

### Q10: 数据安全——上传的文档存哪？API key 怎么管？

**答案骨架**：文档存在 `backend/app/uploads/`（本地文件系统），向量存在 `backend/app/vector_db/`（FAISS + JSON），会话存在 `backend/app/data/sessions/`。API key 通过 `.env` 文件加载（`config.py:108`），不提交到 git（检查 `.gitignore` 可确认）。传给 DeepSeek 的数据是检索到的文档片段——不是全量文档。路径穿越防护：`_safe_file_path()` 校验文件名不包含 `../` 和路径分隔符。

【证据：`routes.py:68-78`，`config.py:72-73,108`】
